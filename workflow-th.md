# เอกสารการทำงานของ GitHub Actions Workflow

## ภาพรวม
เอกสารนี้อธิบายรายละเอียดการทำงานของ GitHub Actions workflows ที่ใช้ในโปรเจ็กต์ทดสอบ SauceDemo Playwright อย่างครบถ้วน ระบบท่อการทดสอบอัตโนมัติของเราช่วยให้มั่นใจในคุณภาพของโค้ดและความน่าเชื่อถือของแอปพลิเคชัน

## โครงสร้างไฟล์ Workflow

### 1. Workflow หลักของ Playwright (`playwright.yml`)
- **วัตถุประสงค์**: Workflow ทั่วไปสำหรับการทดสอบ Playwright
- **ตำแหน่ง**: `.github/workflows/playwright.yml`
- **จุดเน้น**: รันการทดสอบ MCP SauceDemo บนเบราว์เซอร์ Chrome

### 2. Workflow เฉพาะ SauceDemo (`saucedemo-tests.yml`)
- **วัตถุประสงค์**: การทดสอบ E2E ของ SauceDemo แบบเฉพาะทาง
- **ตำแหน่ง**: `.github/workflows/saucedemo-tests.yml`
- **จุดเน้น**: การทดสอบที่ปรับปรุงพร้อมรายงานแบบครบถ้วน

## ตัวกระตุ้น Workflow

Workflow ทั้งสองถูกเรียกใช้โดยเหตุการณ์ต่อไปนี้:

### เหตุการณ์ Push
```yaml
push:
  branches: [ main, develop ]
  paths:
    - 'tests/**'
    - 'playwright.config.js'
    - 'package*.json'
```
- **เมื่อไหร่**: เมื่อมีการ push โค้ดไปยัง branch `main` หรือ `develop`
- **เงื่อนไข**: เฉพาะเมื่อมีการเปลี่ยนแปลงไฟล์ทดสอบ, Playwright config, หรือไฟล์ package
- **วัตถุประสงค์**: ให้แน่ใจว่าการทดสอบจะรันเมื่อมีการเปลี่ยนแปลงโค้ดที่เกี่ยวข้อง

### เหตุการณ์ Pull Request
```yaml
pull_request:
  branches: [ main, develop ]
  paths:
    - 'tests/**'
    - 'playwright.config.js'
    - 'package*.json'
```
- **เมื่อไหร่**: เมื่อมีการเปิด Pull Request ไปยัง branch `main` หรือ `develop`
- **วัตถุประสงค์**: ตรวจสอบการเปลี่ยนแปลงก่อนการรวม

### เหตุการณ์ตามเวลา
```yaml
schedule:
  - cron: '0 2 * * *'
```
- **เมื่อไหร่**: ทุกวันเวลา 2:00 น. UTC
- **วัตถุประสงค์**: ตรวจสอบสุขภาพของแอปพลิเคชันเป็นประจำ

### การเรียกใช้ด้วยตนเอง
```yaml
workflow_dispatch:
```
- **เมื่อไหร่**: เรียกใช้ด้วยตนเองจาก GitHub Actions interface
- **วัตถุประสงค์**: การทดสอบตามความต้องการ

## ขั้นตอนการทำงานของ Workflow แบบรายละเอียด

### ขั้นตอนที่ 1: การตั้งค่าสภาพแวดล้อม

#### Checkout Code
```yaml
- name: Checkout code
  uses: actions/checkout@v4
```
- **วัตถุประสงค์**: ดาวน์โหลดโค้ดจาก repository มายัง runner
- **Action**: ใช้ GitHub's official checkout action (v4)
- **ผลลัพธ์**: ไฟล์โปรเจ็กต์ทั้งหมดพร้อมใช้งานใน workflow

#### Setup Node.js
```yaml
- name: Setup Node.js
  uses: actions/setup-node@v4
  with:
    node-version: 'lts/*'
```
- **วัตถุประสงค์**: ติดตั้ง Node.js runtime environment
- **เวอร์ชัน**: เวอร์ชัน LTS (Long Term Support) ล่าสุด
- **เหตุผลของ LTS**: รับประกันเสถียรภาพและความเข้ากันได้

### ขั้นตอนที่ 2: การจัดการ Dependencies

#### ติดตั้ง Dependencies
```yaml
- name: Install dependencies
  run: npm install
```
- **วัตถุประสงค์**: ติดตั้ง dependencies ทั้งหมดของโปรเจ็กต์
- **คำสั่ง**: `npm install` (ไม่ใช่ `npm ci` เนื่องจากปัญหา lock file)
- **Dependencies ที่ติดตั้ง**:
  - Playwright testing framework
  - Test utilities
  - Dependencies อื่นๆ ของโปรเจ็กต์

#### ติดตั้ง Playwright Browsers
```yaml
- name: Install Playwright browsers
  run: npx playwright install chromium --with-deps
```
- **วัตถุประสงค์**: ดาวน์โหลดและติดตั้ง browser binaries
- **เบราว์เซอร์**: เฉพาะ Chromium (Chrome) เพื่อประสิทธิภาพที่ดีที่สุด
- **Flag `--with-deps`**: ติดตั้ง system dependencies ที่เบราว์เซอร์ต้องการ
- **เหตุผล Chrome เท่านั้น**: ลดเวลา workflow และเพิ่มความน่าเชื่อถือ

### ขั้นตอนที่ 3: การรันการทดสอบ

#### รันการทดสอบ
```yaml
- name: Run SauceDemo MCP tests (JavaScript)
  run: npx playwright test tests/mcp-saucedemo.spec.js --project=chromium
  env:
    CI: true
```
- **วัตถุประสงค์**: รัน test suite หลัก
- **ไฟล์ทดสอบ**: `mcp-saucedemo.spec.js` (เวอร์ชัน JavaScript)
- **Browser Project**: Chromium เท่านั้น
- **Environment**: `CI=true` เปิดใช้งานการปรับปรุงเฉพาะ CI

### ขั้นตอนที่ 4: การสร้างสรุปการทดสอบ

#### สร้างสรุปการทดสอบ
```yaml
- name: Generate test summary
  if: always()
  run: |
    echo "## 🎭 MCP SauceDemo Test Results (Chrome)" >> $GITHUB_STEP_SUMMARY
    # ... เนื้อหาสรุปเพิ่มเติม
```
- **เงื่อนไข**: `if: always()` - รันไม่ว่าขั้นตอนก่อนหน้าจะสำเร็จหรือล้มเหลว
- **วัตถุประสงค์**: สร้างสรุปที่จัดรูปแบบใน GitHub Actions interface
- **เนื้อหา**: 
  - รายละเอียดการรันการทดสอบ
  - ข้อมูลเบราว์เซอร์
  - เวลาประทับ
  - Artifacts ที่มีอยู่

### ขั้นตอนที่ 5: การเก็บ Artifacts

#### Video Artifacts
```yaml
- name: Upload test videos
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: saucedemo-videos-chrome
    path: test-results/**/*.webm
    retention-days: 7
```
- **วัตถุประสงค์**: เก็บรักษาวิดีโอการรันการทดสอบ
- **รูปแบบ**: ไฟล์วิดีโอ WebM
- **การเก็บรักษา**: 7 วัน
- **การใช้งาน**: การ debug แบบภาพของการทดสอบที่ล้มเหลว

#### Screenshot Artifacts
```yaml
- name: Upload test screenshots
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: saucedemo-screenshots-chrome
    path: test-results/**/*.png
    retention-days: 7
```
- **วัตถุประสงค์**: จับภาพหน้าจอที่จุดล้มเหลว
- **รูปแบบ**: ภาพ PNG
- **การเก็บรักษา**: 7 วัน
- **การใช้งาน**: ตรวจสอบการล้มเหลวแบบภาพอย่างรวดเร็ว

#### HTML Report Artifacts
```yaml
- name: Upload HTML report
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: saucedemo-report-chrome
    path: playwright-report/
    retention-days: 14
```
- **วัตถุประสงค์**: รายงานการทดสอบแบบโต้ตอบรายละเอียด
- **รูปแบบ**: HTML พร้อม CSS และ JavaScript
- **การเก็บรักษา**: 14 วัน (นานกว่าเนื่องจากความสำคัญ)
- **คุณลักษณะ**: Timeline การทดสอบ, การรันทีละขั้นตอน, สื่อฝังตัว

#### Trace Files
```yaml
- name: Upload trace files
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: saucedemo-traces-chrome
    path: test-results/**/*.zip
    retention-days: 7
```
- **วัตถุประสงค์**: Execution traces รายละเอียดสำหรับการ debugging
- **รูปแบบ**: ไฟล์ trace ที่บีบอัด
- **คุณลักษณะ**: Network logs, console output, DOM snapshots

### ขั้นตอนที่ 6: การแจ้งเตือนและสรุป

#### Notify Results Job
```yaml
notify-results:
  name: Notify Test Results
  needs: saucedemo-mcp-tests
  runs-on: ubuntu-latest
  if: always()
```
- **Dependency**: รอให้ main test job เสร็จสิ้น
- **เงื่อนไข**: รันเสมอ (สำเร็จหรือล้มเหลว)
- **วัตถุประสงค์**: สร้างสรุปการทดสอบแบบครบถ้วน

## การตัดสินใจเพิ่มประสิทธิภาพ Workflow

### ทำไมเฉพาะ Chrome?
1. **ความน่าเชื่อถือ**: Chrome มีพฤติกรรมที่สม่ำเสมอที่สุดใน CI environments
2. **ความเร็ว**: เบราว์เซอร์เดียวลดเวลาการรัน
3. **ความเข้ากันได้**: รองรับคุณลักษณะเว็บสมัยใหม่ได้ดีที่สุด
4. **ประสิทธิภาพทรัพยากร**: ใช้หน่วยความจำและ CPU น้อยกว่า

### ทำไม npm install vs npm ci?
1. **ปัญหา Lock File**: ปัญหาการสร้าง package-lock.json ใน CI
2. **ความยืดหยุ่น**: อนุญาตให้แก้ไข dependency ในกรณีที่มีข้อขัดแย้ง
3. **ความเข้ากันได้**: จัดการ optional dependencies ได้ดีกว่า

### กลยุทธ์การเก็บรักษา Artifact
- **วิดีโอ/ภาพหน้าจอ**: 7 วัน (สำหรับการ debug ทันที)
- **รายงาน HTML**: 14 วัน (สำหรับการวิเคราะห์แนวโน้ม)
- **Traces**: 7 วัน (การ debug รายละเอียดเมื่อจำเป็น)

## การแก้ไขปัญหาทั่วไป

### การรันการทดสอบล้มเหลว
1. **ตรวจสอบ Artifacts**: ดาวน์โหลดวิดีโอและภาพหน้าจอ
2. **ตรวจสอบรายงาน HTML**: ตรวจสอบขั้นตอนการรันรายละเอียด
3. **วิเคราะห์ Traces**: มองหา network หรือ console errors

### ความล้มเหลวในการติดตั้งเบราว์เซอร์
1. **ตรวจสอบ System Dependencies**: ให้แน่ใจว่าใช้ flag `--with-deps`
2. **ตรวจสอบเวอร์ชัน Node.js**: ยืนยันความเข้ากันได้กับ LTS
3. **ปัญหาเครือข่าย**: ตรวจสอบว่าการดาวน์โหลดเบราว์เซอร์ถูกบล็อกหรือไม่

### ความล้มเหลวในการอัปโหลด Artifact
1. **การตรวจสอบเส้นทาง**: ให้แน่ใจว่าเส้นทาง artifact มีอยู่
2. **สิทธิ์ไฟล์**: ตรวจสอบสิทธิ์การเข้าถึงไฟล์
3. **ขีดจำกัดขนาด**: ตรวจสอบว่า artifacts ไม่เกินขีดจำกัดของ GitHub

## เมตริกประสิทธิภาพ

### เวลาการรันโดยทั่วไป
- **ขั้นตอนการตั้งค่า**: 2-3 นาที
- **การรันการทดสอบ**: 3-5 นาที
- **การอัปโหลด Artifact**: 1-2 นาที
- **ระยะเวลารวม**: 6-10 นาที

### การใช้ทรัพยากร
- **หน่วยความจำ**: ~2GB peak usage
- **CPU**: ใช้ 2 cores
- **พื้นที่เก็บข้อมูล**: ~500MB สำหรับ artifacts

## แนวปฏิบัติที่ดีที่ใช้

### 1. การรันแบบมีเงื่อนไข
- ใช้ `if: always()` สำหรับขั้นตอนการทำความสะอาด
- การเรียกใช้แบบ path-based เพื่อหลีกเลี่ยงการรันที่ไม่จำเป็น

### 2. การบันทึกแบบครบถ้วน
- ชื่อและคำอธิบายขั้นตอนที่มีรายละเอียด
- Console output สำหรับการ debugging

### 3. การจัดการ Artifact
- การตั้งชื่อแบบมีระเบียบ
- ระยะเวลาการเก็บรักษาที่เหมาะสม
- ประเภท artifact หลายแบบสำหรับการใช้งานที่แตกต่างกัน

### 4. การจัดการข้อผิดพลาด
- การจัดการความล้มเหลวอย่างสง่างาม
- ขั้นตอนการทำความสะอาดที่รันเสมอ
- การรายงานข้อผิดพลาดแบบครบถ้วน

## การปรับปรุงในอนาคต

### การปรับปรุงที่วางแผนไว้
1. **การรันการทดสอบแบบขนาน**: รันไฟล์ทดสอบหลายไฟล์พร้อมกัน
2. **การทดสอบข้ามเบราว์เซอร์**: เพิ่ม Firefox และ Safari เมื่อจำเป็น
3. **การทดสอบประสิทธิภาพ**: เพิ่มเมตริกโหลดและประสิทธิภาพ
4. **การทดสอบการรวม**: การตรวจสอบ API endpoint
5. **การทดสอบ Visual Regression**: การทดสอบเปรียบเทียบภาพหน้าจอ

### การปรับปรุงการติดตาม
1. **การแจ้งเตือน Slack**: แจ้งเตือนทีมเมื่อล้มเหลว
2. **แดชบอร์ดเมตริกการทดสอบ**: การวิเคราะห์แนวโน้มประวัติศาสตร์
3. **การตรวจจับการทดสอบที่ไม่เสถียร**: ระบุการทดสอบที่ไม่เสถียร
4. **รายงานครอบคลุม**: การวิเคราะห์ครอบคลุมโค้ด

GitHub Actions workflows ให้บริการท่อการทดสอบอัตโนมัติที่แข็งแกร่ง ที่รับประกันคุณภาพของโค้ดและความน่าเชื่อถือของแอปพลิเคชัน การรันทีละขั้นตอน การเก็บ artifact แบบครบถ้วน และการรายงานรายละเอียดทำให้ง่ายต่อการระบุและแก้ไขปัญหาอย่างรวดเร็ว

Workflows ได้รับการปรับปรุงเพื่อความน่าเชื่อถือและความเร็วในขณะที่รักษาการครอบคลุมการทดสอบแบบครบถ้วนของเส้นทางผู้ใช้ที่สำคัญของแอปพลิเคชัน SauceDemo
