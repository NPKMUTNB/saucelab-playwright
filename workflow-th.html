🎯 ภาพรวม

เอกสารนี้อธิบายการทำงานของ GitHub Actions workflows ที่ใช้ในโปรเจ็กต์ทดสอบ SauceDemo Playwright อย่างครบถ้วน ท่อการทดสอบอัตโนมัติของเราช่วยให้มั่นใจในคุณภาพของโค้ดและความน่าเชื่อถือของแอปพลิเคชัน

🎪 สิ่งที่เอกสารนี้ครอบคลุม

  • กระบวนการรัน workflow ทีละขั้นตอน
  • เงื่อนไขการเรียกใช้และกฎการทำงานอัตโนมัติ
  • กลยุทธ์การเก็บและการเก็บรักษา artifacts
  • การแก้ไขปัญหาทั่วไป
  • เทคนิคการปรับปรุงประสิทธิภาพ

📁 โครงสร้างไฟล์ Workflow

⚡Workflow หลักของ Playwright

ไฟล์: playwright.yml

วัตถุประสงค์: Workflow ทั่วไปสำหรับการทดสอบ Playwright

จุดเน้น: รันการทดสอบ MCP SauceDemo บนเบราว์เซอร์ Chrome

🎯Workflow เฉพาะ SauceDemo

ไฟล์: saucedemo-tests.yml

วัตถุประสงค์: การทดสอบ E2E ของ SauceDemo แบบเฉพาะทาง

จุดเน้น: การทดสอบที่ปรับปรุงพร้อมรายงานแบบครบถ้วน

🚀 ตัวกระตุ้น Workflow

1เหตุการณ์ Push

push: branches: [ main, develop ] paths: - 'tests/**' - 'playwright.config.js' - 'package*.json'

เมื่อไหร่: เมื่อมีการ push โค้ดไปยัง branch main หรือ develop พร้อมการเปลี่ยนแปลงที่เกี่ยวข้องกับการทดสอบ

2เหตุการณ์ Pull Request

pull_request: branches: [ main, develop ]

วัตถุประสงค์: ตรวจสอบการเปลี่ยนแปลงก่อนการรวม

3เหตุการณ์ตามเวลา

schedule: - cron: '0 2 * * *'

เมื่อไหร่: ทุกวันเวลา 2:00 น. UTC สำหรับการตรวจสอบสุขภาพเป็นประจำ

4การเรียกใช้ด้วยตนเอง

workflow_dispatch:

วัตถุประสงค์: การทดสอบตามความต้องการจาก GitHub Actions interface

🔧 ขั้นตอนการทำงานของ Workflow แบบรายละเอียด

1การตั้งค่าสภาพแวดล้อม

Checkout Code & Setup Node.js

- name: Checkout code uses: actions/checkout@v4 - name: Setup Node.js uses: actions/setup-node@v4 with: node-version: 'lts/*'

ดาวน์โหลดโค้ดจาก repository และติดตั้ง Node.js เวอร์ชัน LTS ล่าสุดเพื่อความเสถียร

2การจัดการ Dependencies

- name: Install dependencies run: npm install - name: Install Playwright browsers run: npx playwright install chromium --with-deps

ติดตั้ง dependencies ของโปรเจ็กต์และเบราว์เซอร์ Chrome พร้อม system dependencies

3การรันการทดสอบ

- name: Run SauceDemo MCP tests run: npx playwright test tests/mcp-saucedemo.spec.js --project=chromium env: CI: true

รัน test suite หลักพร้อมการปรับปรุงเฉพาะ CI ที่เปิดใช้งาน

4การเก็บ Artifacts

🎥วิดีโอ

รูปแบบ WebM, เก็บรักษา 7 วัน

📸ภาพหน้าจอ

รูปแบบ PNG, เก็บรักษา 7 วัน

📊รายงาน HTML

รายงานแบบโต้ตอบ, เก็บรักษา 14 วัน

🔍ไฟล์ Trace

Debug traces, เก็บรักษา 7 วัน

  • การตรวจสอบราคา

🛒การจัดการตะกร้าสินค้า

  • การอัปเดต badge ตะกร้า
  • การจัดการหลายสินค้า

จุดตรวจสอบหลักคือการยืนยันข้อความ "Thank you for your order!" ซึ่งยืนยันว่ากระบวนการซื้อแบบครบวงจรทำงานได้อย่างถูกต้อง

🔧 การแก้ไขปัญหาทั่วไป

1การรันการทดสอบล้มเหลว

  • ดาวน์โหลดและตรวจสอบ artifacts วิดีโอและภาพหน้าจอ
  • ตรวจสอบรายงาน HTML สำหรับขั้นตอนการรันรายละเอียด
  • วิเคราะห์ trace files สำหรับ network หรือ console errors

2ปัญหาการติดตั้งเบราว์เซอร์

  • ให้แน่ใจว่าใช้ flag --with-deps
  • ตรวจสอบความเข้ากันได้กับ Node.js LTS
  • ตรวจสอบปัญหาการเชื่อมต่อเครือข่าย

3ปัญหาการอัปโหลด Artifact

  • ตรวจสอบว่าเส้นทาง artifact ไฟล์มีอยู่
  • ตรวจสอบสิทธิ์และการเข้าถึงไฟล์
  • ตรวจสอบว่า artifacts ไม่เกินขีดจำกัดของ GitHub

🚀 การพัฒนาในอนาคต

การปรับปรุงที่วางแผนไว้

  • การรันแบบขนาน: รันไฟล์ทดสอบหลายไฟล์พร้อมกัน
  • การทดสอบข้ามเบราว์เซอร์: รองรับ Firefox และ Safari
  • การทดสอบประสิทธิภาพ: เมตริกโหลดและประสิทธิภาพ
  • การทดสอบ Visual Regression: การเปรียบเทียบภาพหน้าจอ
  • แดชบอร์ดการติดตาม: การวิเคราะห์แนวโน้มประวัติศาสตร์

คุณลักษณะการติดตาม

  • การแจ้งเตือน Slack: แจ้งเตือนทีมเมื่อล้มเหลว
  • การตรวจจับการทดสอบที่ไม่เสถียร: ระบุการทดสอบที่มีปัญหา
  • รายงานความครอบคลุม: การวิเคราะห์ครอบคลุมโค้ด

📝 สรุป

GitHub Actions workflows ให้บริการท่อการทดสอบอัตโนมัติที่แข็งแกร่ง ที่รับประกันคุณภาพของโค้ดและความน่าเชื่อถือของแอปพลิเคชัน การรันทีละขั้นตอน การเก็บ artifact แบบครบถ้วน และการรายงานรายละเอียดทำให้ง่ายต่อการระบุและแก้ไขปัญหาอย่างรวดเร็ว
