# เอกสารการกำหนดค่า Dependabot

## ภาพรวม

Dependabot เป็นเครื่องมือจัดการ dependencies อัตโนมัติของ GitHub ที่ช่วยให้ dependencies ของโปรเจ็กต์ของคุณอัปเดตอยู่เสมอ โดยการสร้าง pull requests อัตโนมัติเมื่อมีเวอร์ชันใหม่ออกมา เอกสารนี้จะอธิบายการทำงานของไฟล์การกำหนดค่า `dependabot.yml` และวิธีการจัดการ dependencies ในโปรเจ็กต์ทดสอบ Playwright นี้

## Dependabot คืออะไร?

Dependabot เป็นบริการของ GitHub ที่:
- **ตรวจสอบ Dependencies**: สแกนโปรเจ็กต์ของคุณเพื่อหา dependencies ที่ล้าสมัย
- **สร้าง Pull Requests**: สร้าง PRs พร้อมการอัปเดต dependencies เมื่อมีเวอร์ชันใหม่
- **อัปเดตความปลอดภัย**: ให้ความสำคัญกับการแก้ไขช่องโหว่ด้านความปลอดภัย
- **การกำหนดเวลาที่ปรับแต่งได้**: ให้คุณควบคุมว่าเมื่อไหร่และอย่างไรที่จะเสนอการอัปเดต
- **รองรับหลาย Ecosystems**: รองรับ package managers ต่างๆ (npm, Maven, pip, ฯลฯ)

## โครงสร้างไฟล์การกำหนดค่า

การกำหนดค่า Dependabot ของเราอยู่ที่ `.github/dependabot.yml` และมีโครงสร้างดังนี้:

```yaml
version: 2
updates:
  # การกำหนดค่าสำหรับ package ecosystems ต่างๆ
```

### การประกาศเวอร์ชัน
```yaml
version: 2
```
ระบุว่าเราใช้ Dependabot configuration schema เวอร์ชัน 2 ซึ่งเป็นเวอร์ชันปัจจุบันและแนะนำ

## การกำหนดค่า Package Ecosystem

### 1. การกำหนดค่า NPM Ecosystem

```yaml
- package-ecosystem: "npm"
  directory: "/"
  schedule:
    interval: "weekly"
    day: "monday"
    time: "09:00"
  open-pull-requests-limit: 10
  reviewers:
    - "dependabot[bot]"
  assignees:
    - "dependabot[bot]"
  commit-message:
    prefix: "⬆️"
    include: "scope"
```

#### รายละเอียดการกำหนดค่า:

- **`package-ecosystem: "npm"`**: กำหนดเป้าหมายเป็น NPM packages (dependencies ของ JavaScript/Node.js)
- **`directory: "/"`**: ตรวจสอบไดเรกทอรีรากสำหรับ `package.json` และ `package-lock.json`
- **`schedule`**: กำหนดเวลาที่ Dependabot จะตรวจสอบการอัปเดต
  - **`interval: "weekly"`**: ตรวจสอบการอัปเดตทุกสัปดาห์
  - **`day: "monday"`**: ทำการตรวจสอบทุกวันจันทร์
  - **`time: "09:00"`**: ดำเนินการเวลา 9:00 น. UTC
- **`open-pull-requests-limit: 10`**: จำกัด PRs พร้อมกันสูงสุด 10 รายการสำหรับ npm updates
- **`reviewers`**: กำหนด `dependabot[bot]` เป็น reviewer อัตโนมัติ
- **`assignees`**: กำหนด `dependabot[bot]` เป็น assignee อัตโนมัติ
- **`commit-message`**: ปรับแต่ง commit messages
  - **`prefix: "⬆️"`**: เพิ่ม emoji ลูกศรเพื่อระบุการอัปเดต dependency
  - **`include: "scope"`**: รวม scope ใน commit messages

### 2. การกำหนดค่า GitHub Actions Ecosystem

```yaml
- package-ecosystem: "github-actions"
  directory: "/"
  schedule:
    interval: "weekly"
    day: "monday"
    time: "09:00"
  open-pull-requests-limit: 5
  reviewers:
    - "dependabot[bot]"
  assignees:
    - "dependabot[bot]"
  commit-message:
    prefix: "🔧"
    include: "scope"
```

#### รายละเอียดการกำหนดค่า:

- **`package-ecosystem: "github-actions"`**: กำหนดเป้าหมายเป็น GitHub Actions workflows
- **`directory: "/"`**: ตรวจสอบไดเรกทอรี `.github/workflows/` สำหรับไฟล์ workflow
- **`schedule`**: กำหนดเวลาเดียวกัน คือ จันทร์ 9:00 น. UTC ทุกสัปดาห์
- **`open-pull-requests-limit: 5`**: จำกัด PRs พร้อมกันสูงสุด 5 รายการสำหรับ GitHub Actions updates
- **`commit-message`**: ใช้ emoji 🔧 (ประแจ) เป็น prefix สำหรับ Actions updates

## วิธีการทำงานของ Dependabot

### 1. การสแกน Dependencies
- **NPM Packages**: สแกน `package.json` และ `package-lock.json` เพื่อหา dependencies ที่ล้าสมัย
- **GitHub Actions**: ตรวจสอบไฟล์ workflow (`.yml`/`.yaml`) ในไดเรกทอรี `.github/workflows/`
- **การเปรียบเทียบเวอร์ชัน**: เปรียบเทียบเวอร์ชันปัจจุบันกับเวอร์ชันล่าสุดที่มีอยู่

### 2. การสร้าง Pull Request
เมื่อมีการอัปเดต, Dependabot จะ:
1. **สร้าง Branch**: สร้าง branch ใหม่พร้อมการอัปเดต dependency
2. **อัปเดตไฟล์**: แก้ไข `package.json`, `package-lock.json`, หรือไฟล์ workflow
3. **สร้าง PR**: เปิด pull request พร้อมข้อมูลรายละเอียด
4. **รัน Tests**: เรียกใช้ CI/CD workflows เพื่อตรวจสอบการอัปเดต

### 3. เนื้อหา PR
Dependabot PR แต่ละรายการจะรวมถึง:
- **ข้อมูล Changelog**: สิ่งใหม่ในเวอร์ชันที่อัปเดต
- **คะแนนความเข้ากันได้**: ระดับความมั่นใจสำหรับการอัปเดต
- **Release Notes**: ลิงก์ไปยัง release notes และเอกสาร
- **ประวัติ Commit**: การเปลี่ยนแปลงระหว่างเวอร์ชัน
- **ข้อมูลช่องโหว่**: การแก้ไขความปลอดภัยหากมี

## กำหนดการและเวลา

### ประโยชน์ของกำหนดการรายสัปดาห์
- **การอัปเดตที่คาดเดาได้**: ทีมรู้ว่าเมื่อไหร่จะมี dependency PRs
- **ปริมาณที่จัดการได้**: ช่วงเวลารายสัปดาห์ป้องกันไม่ให้ทีมครอบงำ
- **เวลาวันจันทร์**: ให้จัดการการอัปเดตที่จุดเริ่มต้นของสัปดาห์การทำงาน
- **UTC 9:00 น.**: โดยทั่วไปตรงกับเวลาทำการทั่วโลก

### ขีดจำกัด Pull Request
- **ขีดจำกัด NPM (10)**: ขีดจำกัดสูงกว่าเนื่องจาก JavaScript dependency updates บ่อย
- **ขีดจำกัด GitHub Actions (5)**: ขีดจำกัดต่ำกว่าเนื่องจาก Actions updates น้อยกว่า
- **ป้องกันการโอเวอร์โหลด**: ขีดจำกัดป้องกันไม่ให้มี PRs พร้อมกันมากเกินไป

## แนวทางการตั้งชื่อ Commit Message

```

```

### GitHub Actions Updates
```

```

### ประโยชน์ของ Prefixes
- **การระบุด้วยภาพ**: ง่ายต่อการสังเกตการอัปเดต dependency ในประวัติ commit
- **การกรอง**: สามารถกรอง commits ด้วย emoji เพื่อดูเฉพาะการอัปเดต dependency
- **ระบบอัตโนมัติ**: ระบบ CI/CD สามารถจัดการ commits เหล่านี้แตกต่างออกไป

## การจัดการความปลอดภัยและช่องโหว่

### การอัปเดตความปลอดภัยอัตโนมัติ
Dependabot ให้ความสำคัญกับการอัปเดตความปลอดภัยโดย:
- **การแจ้งเตือนทันที**: สร้าง PRs สำหรับช่องโหว่ความปลอดภัยไม่ว่าจะเป็นกำหนดการใด
- **ความสำคัญสูง**: การอัปเดตความปลอดภัยข้ามกำหนดการปกติ
- **การติดป้ายที่ชัดเจน**: PRs ความปลอดภัยถูกทำเครื่องหมายและจัดทำเอกสารอย่างชัดเจน

### ข้อมูลช่องโหว่
Security PR แต่ละรายการรวมถึง:
- **ข้อมูล CVE**: รายละเอียด Common Vulnerabilities and Exposures
- **ระดับความรุนแรง**: ความรุนแรง Critical, High, Medium, หรือ Low
- **คะแนน CVSS**: การให้คะแนน Common Vulnerability Scoring System
- **การแก้ไข**: วิธีที่การอัปเดตแก้ไขช่องโหว่

## การผสานรวม Workflow

### CI/CD Pipeline
Dependabot PRs ของเราเรียกใช้อัตโนมัติ:
1. **Playwright Tests**: รันชุดทดสอบเพื่อให้แน่ใจว่าเข้ากันได้
2. **GitHub Actions Workflows**: ตรวจสอบว่า actions ที่อัปเดตทำงานถูกต้อง
3. **กระบวนการ Build**: ให้แน่ใจว่าโปรเจ็กต์ build สำเร็จด้วย dependencies ใหม่

### กระบวนการ Review
1. **การตรวจสอบอัตโนมัติ**: CI/CD pipeline ตรวจสอบการอัปเดต
2. **การ Review ด้วยตนเอง**: ทีมตรวจสอบ PR สำหรับปัญหาที่อาจเกิดขึ้น
3. **การทดสอบ**: การทดสอบเพิ่มเติมหากจำเป็นสำหรับการอัปเดตหลัก
4. **การรวม**: การอัปเดตที่ได้รับอนุมัติจะถูกรวมเข้าสู่ main branch

## แนวปฏิบัติที่ดี

### การจัดการ Dependabot PRs
1. **การ Review เป็นประจำ**: ตรวจสอบ PRs รายสัปดาห์ในช่วงเวลาที่กำหนด
2. **ทดสอบอย่างละเอียด**: รันการทดสอบเพิ่มเติมสำหรับการอัปเดตเวอร์ชันหลัก
3. **การจัดกลุ่มการอัปเดต**: พิจารณาการจัดกลุ่มการอัปเดต dependency ที่เกี่ยวข้อง
4. **ตรวจสอบผลกระทบ**: เฝ้าระวังปัญหาใดๆ หลังจากรวมการอัปเดต

### การปรับปรุงการกำหนดค่า
- **ปรับขีดจำกัด**: แก้ไขขีดจำกัด PR ตามความสามารถของทีม
- **เวลากำหนดการ**: เลือกเวลาที่เหมาะสมสำหรับ workflow ของทีม
- **การเลือก Ecosystem**: เปิดใช้งานเฉพาะ package ecosystems ที่เกี่ยวข้อง

## การแก้ไขปัญหา

### ปัญหาทั่วไป

#### PRs มากเกินไป
**ปัญหา**: ครอบงำด้วยการอัปเดต dependency
**แก้ไข**: ลด `open-pull-requests-limit` หรือเปลี่ยนเป็นกำหนดการรายเดือน

#### การอัปเดตล้มเหลว
**ปัญหา**: Dependabot PRs ล้มเหลวในการตรวจสอบ CI/CD
**แก้ไข**: ตรวจสอบการเปลี่ยนแปลงที่ทำลายใน changelog และอัปเดตโค้ดตามที่จำเป็น

#### การอัปเดตที่หายไป
**ปัญหา**: การอัปเดตที่คาดหวังไม่ปรากฏ
**แก้ไข**: ตรวจสอบ Dependabot logs ใน repository insights

### การตรวจสอบการกำหนดค่า
- **ไวยากรณ์ YAML**: ให้แน่ใจว่ามีการจัดรูปแบบ YAML ที่เหมาะสม
- **ตำแหน่งไฟล์**: ตรวจสอบว่าไฟล์อยู่ที่ `.github/dependabot.yml`
- **สิทธิ์**: ให้แน่ใจว่า Dependabot มีสิทธิ์เข้าถึง repository ที่เหมาะสม

## ประโยชน์สำหรับโปรเจ็กต์นี้

### สำหรับการทดสอบ Playwright
- **Browser Drivers ที่อัปเดต**: รับประกันความเข้ากันได้ของเบราว์เซอร์ล่าสุด
- **การแก้ไขความปลอดภัย**: รักษา testing dependencies ให้ปลอดภัย
- **การปรับปรุงประสิทธิภาพ**: ได้รับประโยชน์จากการปรับปรุงประสิทธิภาพใน dependencies
- **การแก้ไขบัค**: รับการแก้ไขบัคใน testing frameworks อัตโนมัติ

### สำหรับ GitHub Actions
- **คุณลักษณะล่าสุด**: ได้รับประโยชน์จากคุณลักษณะ GitHub Actions ใหม่
- **การอัปเดตความปลอดภัย**: รักษา CI/CD pipeline ให้ปลอดภัย
- **ประสิทธิภาพ**: ปรับปรุงเวลาการดำเนินการ workflow
- **ความเข้ากันได้**: รักษาความเข้ากันได้กับการอัปเดตแพลตฟอร์ม GitHub

## การตรวจสอบและเมตริก

### การติดตามกิจกรรม Dependabot
- **แท็บ Insights**: ดูกิจกรรม Dependabot ใน repository insights
- **ประวัติ PR**: ติดตามการอัปเดตที่สำเร็จและล้มเหลว
- **การแจ้งเตือนความปลอดภัย**: ตรวจสอบการแก้ไขช่องโหว่ความปลอดภัย
- **กราฟ Dependency**: แสดงภาพ dependencies ของโปรเจ็กต์

### เมตริกความสำเร็จ
- **ความถี่การอัปเดต**: ความถี่ในการอัปเดต dependencies
- **เวลาตอบสนองความปลอดภัย**: เวลาในการจัดการช่องโหว่ความปลอดภัย
- **อัตราความสำเร็จ CI/CD**: เปอร์เซ็นต์ของการอัปเดตอัตโนมัติที่สำเร็จ
- **การแทรกแซงด้วยตนเอง**: ความถี่ที่ต้องแก้ไขด้วยตนเอง

## สรุป

การกำหนดค่า Dependabot ของเราให้การจัดการ dependency อัตโนมัติและตามกำหนดการที่:
- **เพิ่มความปลอดภัย**: ใช้การแก้ไขความปลอดภัยอัตโนมัติ
- **ลดหนี้ทางเทคนิค**: ป้องกันไม่ให้ dependencies ล้าสมัยอย่างรุนแรง
- **ประหยัดเวลา**: ทำให้กระบวนการตรวจสอบการอัปเดตด้วยตนเองเป็นอัตโนมัติ
- **ปรับปรุงความน่าเชื่อถือ**: รักษาโปรเจ็กต์ให้ทำงานบน dependencies ที่ได้รับการสนับสนุนและบำรุงรักษา
